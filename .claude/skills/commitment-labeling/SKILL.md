# Commitment labeling convention

A commitment is recallable when its filename, ID, and index entry are all consistent. This skill specifies the format so any subagent (or human) can find what they need without spelunking.

## Folder layout

```
/Augurian Clients/[Client]/processed/commitments/
├── _index.jsonl                   # append-only flat index, one item per line
├── 2026-05-04-monthly-review.json # per-call extract from Fireflies
├── 2026-05-07-mike-followup-email.json   # per-email extract
├── 2026-04-coborns-onboarding-intake.json # per-onboarding-doc extract
├── 2026-05/                       # human-readable monthly views
│   ├── upcoming.md                # generated; never hand-edited
│   └── overdue.md                 # generated
└── _archive/                      # finished/cancelled items moved here quarterly
```

## Per-call / per-source filename

```
{YYYY-MM-DD}-{call-or-source-slug}.json
```

- `YYYY-MM-DD` is the **captured date** (when the call happened or the email was sent).
- `call-or-source-slug` is short, lowercase, hyphenated. Examples:
  - `2026-05-04-monthly-review.json`
  - `2026-05-07-mike-followup-email.json`
  - `2026-04-onboarding-intake.json`
- No client name in the filename — the folder already says which client.

## Item ID format

```
{client-slug}-{YYYY-MM-DD}-{NNN}
```

- `NNN` is a zero-padded sequence within that day's source. So a call that produces 5 items gets `001`–`005`.
- Deterministic — re-running the extractor on the same source produces the same IDs.
- Example: `coborns-2026-05-04-003`.

## Index schema (JSONL — one item per line)

Append-only. Never rewrite. Never modify.

```json
{
  "id": "coborns-2026-05-04-001",
  "client": "coborns",
  "type": "deliverable",
  "captured_date": "2026-05-04",
  "due_date": "2026-05-11",
  "owner": "Sarah",
  "owner_role": "augurian",
  "priority": 2,
  "status": "open",
  "tags": ["seo", "q3-planning"],
  "source_path": "processed/commitments/2026-05-04-monthly-review.json",
  "deliverable_summary": "Q3 SEO brief"
}
```

Keep the index entry small (~300 bytes) — it's read every query, so size matters. Full text + verbatim live in the per-call JSON file.

### Status updates

Status changes (`open` → `done` / `cancelled`) **append a new row**, don't modify in place:

```json
{"id":"coborns-2026-05-04-001","status":"done","status_changed_at":"2026-05-11T14:00:00Z","status_changed_by":"sarah","note":"Brief sent to Mike."}
```

The query agents (commitment-tracker, etc.) read the index newest-to-oldest and apply the most recent status. The audit trail is preserved.

## Generated views

`/processed/commitments/<YYYY-MM>/upcoming.md` — generated, not hand-edited. Format:

```markdown
# Upcoming deliverables — {Client} — {YYYY-MM}

Last refreshed: {timestamp}. Source: `_index.jsonl`.

## Open, due this month

| Due | Type | Item | Owner | Priority | Source |
|---|---|---|---|---|---|
| 2026-05-11 | deliverable | Q3 SEO brief | Sarah | 2 | 2026-05-04 monthly review |
| 2026-05-15 | action_item | Pull GSC data | Sarah | 2 | 2026-05-04 monthly review |
| 2026-05-30 | commitment | Hold paid spend flat | Augurian | 1 | 2026-05-04 monthly review |

## Overdue

(none)

## Open, no due date

| Type | Item | Captured | Source |
|---|---|---|---|
| open_question | YouTube expansion | 2026-05-04 | 2026-05-04 monthly review |
```

Generated by `commitment-tracker` on demand or as a Phase 4+ daily job.

## Tag vocabulary

Same vocabulary as the `fireflies-extraction-rules` skill — keep them aligned. If a new tag is needed, the drive-data-architect approves it before use; otherwise tag drift breaks the index.

## What the structure enables (the query path)

A user asks: *"What were the top deliverables for Coborn's for next month?"*

```
1. Resolve "Coborn's" → slug = "coborns"
2. Resolve "next month" → 2026-06-01 to 2026-06-30
3. Read /Augurian Clients/Coborn's/processed/commitments/_index.jsonl
4. Filter: client=coborns, type∈{deliverable,action_item},
           due_date ∈ [2026-06-01, 2026-06-30], status=open
5. Sort: priority desc, due_date asc
6. Return top 5 with deliverable_summary, owner, due_date, source
```

That's it. No vector DB. No fuzzy search. No inference layer. The naming convention IS the query interface.

## Anti-patterns

- **Status update by overwriting.** Breaks audit trail. Always append.
- **Cross-client items.** A commitment lives in exactly one client's folder. If a call covered three clients, write three per-call JSON files (one per client).
- **Filename with apostrophes / spaces.** Use slugs. `coborns`, not `Coborn's`.
- **Tags invented at write time.** Coordinate via `drive-data-architect` before introducing a new tag.
- **Hand-editing generated views.** They get regenerated and overwritten. If a human needs to add notes, do it in the per-call JSON's free-text field, not the markdown view.
